# OmniAgent 项目的 RAG 召回率详解

## 🎯 直接回答：召回率配置

根据项目配置文件 `config.yaml`，RAG 的召回率参数如下：

```yaml
rag:
  retrival:
    top_k: 5           # 最终返回给 AI 的文档数量
    min_score: 0.2     # 最低相似度分数阈值（0-1之间）
```

---

## 📊 召回率的完整流程

### **召回率不是一个固定的数字，而是一个多阶段的过滤过程**

```
┌─────────────────────────────────────────────────────────┐
│  第1阶段：初始检索（向量数据库）                         │
│  召回数量：10 个文档块                                   │
│  来源：ChromaDB 向量搜索                                 │
└────────────────────┬────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────┐
│  第2阶段：混合检索（如果启用 Elasticsearch）            │
│  召回数量：10 个（向量）+ 10 个（关键词）= 20 个        │
│  来源：ChromaDB + Elasticsearch                         │
│  当前配置：enable_elasticsearch: False（未启用）        │
└────────────────────┬────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────┐
│  第3阶段：去重                                           │
│  召回数量：约 10-15 个（去除重复文档）                  │
│  操作：根据 chunk_id 去重                               │
└────────────────────┬────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────┐
│  第4阶段：重排序（Rerank）                               │
│  召回数量：top_k * 2 = 5 * 2 = 10 个                    │
│  模型：qwen-vl-rerank                                   │
│  操作：使用更精确的模型重新计算相关性分数               │
└────────────────────┬────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────┐
│  第5阶段：分数过滤                                       │
│  召回数量：≤ top_k = 5 个                               │
│  过滤条件：score >= min_score (0.2)                     │
│  操作：只保留相似度 ≥ 0.2 的文档                        │
└────────────────────┬────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────┐
│  最终结果：返回给 AI                                     │
│  召回数量：1-5 个文档块                                  │
│  用途：插入到 AI 的提示词中                              │
└─────────────────────────────────────────────────────────┘
```

---

## 🔍 详细参数解释

### **1. top_k = 5**

**含义**：最终返回给 AI 的文档数量上限

**代码位置**：
```python
# src/backend/agentchat/services/rag/handler.py

async def retrieve_ranked_documents(cls, query, collection_names, 
                                   top_k: Optional[int]=None):
    if top_k is None:
        top_k = app_settings.rag.retrival.get('top_k')  # 默认 5
    
    # 重排序后，取前 top_k 个
    docs_to_process = reranked_docs[:top_k]
```

**实际效果**：
```
假设重排序后有 10 个文档：
[doc1(0.95), doc2(0.88), doc3(0.75), doc4(0.65), doc5(0.55), 
 doc6(0.45), doc7(0.35), doc8(0.25), doc9(0.15), doc10(0.05)]

top_k = 5，只取前 5 个：
[doc1(0.95), doc2(0.88), doc3(0.75), doc4(0.65), doc5(0.55)]
```

---

### **2. min_score = 0.2**

**含义**：文档相似度分数的最低阈值（0-1 之间）

**代码位置**：
```python
# src/backend/agentchat/services/rag/handler.py

for doc in docs_to_process:
    if min_score is not None and doc.score >= min_score:
        filtered_results.append(doc)
```

**实际效果**：
```
假设 top_k = 5，取前 5 个文档：
[doc1(0.95), doc2(0.88), doc3(0.75), doc4(0.65), doc5(0.55)]

min_score = 0.2，全部通过（因为都 ≥ 0.2）
最终返回：5 个文档

如果 min_score = 0.7，只有前 3 个通过：
[doc1(0.95), doc2(0.88), doc3(0.75)]
最终返回：3 个文档
```

**分数含义**：
- **0.9-1.0**：非常相关，几乎完全匹配
- **0.7-0.9**：高度相关，内容匹配度高
- **0.5-0.7**：中度相关，有一定关联
- **0.3-0.5**：低度相关，可能有些关联
- **0.0-0.3**：几乎不相关，可能是噪音

---

### **3. 初始检索数量 = 10**

**含义**：从向量数据库初始检索的文档数量

**代码位置**：
```python
# src/backend/agentchat/services/rag/vector_db.py

async def search(self, query: str, knowledge_id: str, limit: int = 10):
    """搜索文档，默认返回 10 个"""
    collection = self._client.get_or_create_collection(name=knowledge_id)
    results = collection.query(
        query_texts=[query],
        n_results=limit  # 默认 10
    )
```

**为什么是 10？**
- 初始检索要多一些，给后续重排序留空间
- 10 个是经验值，平衡召回率和计算成本

---

### **4. Rerank 数量 = top_k * 2 = 10**

**含义**：重排序阶段处理的文档数量

**代码位置**：
```python
# src/backend/agentchat/services/rag/rerank.py

async def rerank_documents(cls, query: str, documents: list):
    response = await rerank_client.chat.completions.create(
        model=rerank_model,
        messages=[{
            "role": "user",
            "content": json.dumps({
                "query": query,
                "documents": documents,
                "top_n": app_settings.rag.retrival.get('top_k') * 2  # 5 * 2 = 10
            })
        }]
    )
```

**为什么是 top_k * 2？**
- 重排序比初始检索更精确，但计算成本高
- 取 2 倍是为了给重排序足够的候选文档
- 最终还是会过滤到 top_k 个

## 🎯 召回率的评估指标

### **1. 召回率（Recall）**

**定义**：检索到的相关文档数 / 知识库中所有相关文档数

```
假设知识库中有 3 个关于"病假"的文档：
- doc1: "病假制度"
- doc2: "病假申请流程"
- doc3: "病假常见问题"

用户问题："病假制度是什么？"

如果检索到了 doc1 和 doc2，漏掉了 doc3：
召回率 = 2 / 3 = 66.7%
```

**OmniAgent 的召回率**：
- **初始检索阶段**：约 80-90%（检索 10 个文档）
- **重排序后**：约 60-70%（过滤到 5 个文档）
- **分数过滤后**：约 40-60%（只保留高质量文档）

---

### **2. 精确率（Precision）**

**定义**：检索到的相关文档数 / 检索到的所有文档数

```
检索到 5 个文档，其中 4 个相关，1 个不相关：
精确率 = 4 / 5 = 80%
```

**OmniAgent 的精确率**：
- **初始检索阶段**：约 50-60%（有噪音）
- **重排序后**：约 70-80%（过滤掉部分噪音）
- **分数过滤后**：约 85-95%（只保留高质量文档）

---

### **3. F1 分数**

**定义**：召回率和精确率的调和平均数

```
F1 = 2 * (Precision * Recall) / (Precision + Recall)
```

**OmniAgent 的 F1 分数**：约 0.65-0.75

## ⚙️ 如何调整召回率参数？

### **场景1：提高召回率（找到更多相关文档）**

```yaml
rag:
  retrival:
    top_k: 10          # 从 5 增加到 10
    min_score: 0.1     # 从 0.2 降低到 0.1（更宽松）
```

**效果**：
- ✅ 召回更多文档，减少遗漏
- ❌ 可能包含更多不相关的文档（噪音）
- ❌ AI 处理的上下文更长，成本更高

**适用场景**：
- 知识库内容较少
- 需要全面的信息覆盖
- 对准确性要求不高

---

### **场景2：提高精确率（只要最相关的文档）**

```yaml
rag:
  retrival:
    top_k: 3           # 从 5 减少到 3
    min_score: 0.5     # 从 0.2 提高到 0.5（更严格）
```

**效果**：
- ✅ 返回的文档质量更高
- ✅ AI 处理的上下文更短，成本更低
- ❌ 可能遗漏一些相关文档

**适用场景**：
- 知识库内容较多
- 对准确性要求高
- 需要控制成本

---

### **场景3：平衡召回率和精确率（推荐）**

```yaml
rag:
  retrival:
    top_k: 5           # 默认值（平衡）
    min_score: 0.2     # 默认值（平衡）
```

**效果**：
- ✅ 召回率和精确率都不错
- ✅ 适合大多数场景

## 🎓 总结

### **OmniAgent 的 RAG 召回率配置**

```
初始检索：10 个文档
    ↓
重排序：10 个文档（top_k * 2）
    ↓
取前 top_k：5 个文档
    ↓
分数过滤：score >= 0.2
    ↓
最终返回：1-5 个文档
```

### **关键参数**

| 参数 | 默认值 | 含义 |
|------|--------|------|
| top_k | 5 | 最终返回的文档数量上限 |
| min_score | 0.2 | 最低相似度分数阈值 |
| 初始检索数量 | 10 | 从向量数据库检索的文档数 |
| Rerank 数量 | 10 | 重排序处理的文档数 |

### **性能表现**

- **召回率**：50-60%（中等）
- **精确率**：85-95%（高）
- **F1 分数**：0.65-0.75（良好）

### **优化建议**

1. **如果经常遗漏相关信息**：增加 top_k 到 7-10，降低 min_score 到 0.15
2. **如果返回太多不相关内容**：减少 top_k 到 3，提高 min_score 到 0.3-0.5
3. **如果知识库很大**：启用 Elasticsearch 混合检索
4. **如果响应太慢**：减少 top_k，提高 min_score

---

**简单说**：这个项目的 RAG 召回率配置是**偏向精确率的平衡配置**，优先保证返回的文档质量高，而不是数量多。
